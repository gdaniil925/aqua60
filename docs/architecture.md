# Архитектура Aqua60

## 1. Контуры системы

### 1.1 Клиент

Telegram Web App внутри бота:

- оферта и согласие на рекуррентные платежи
- регистрация и профиль
- конструктор подписки
- личный кабинет
- дозаказ и управление графиком
- уведомления о доставке и статус экспресс-заказа

### 1.2 Курьер

Telegram Web App или выделенный web-интерфейс:

- статус смены
- очередь плановых и экспресс-заказов
- карточка заказа
- подтверждение принятия заказа
- завершение доставки
- виртуальный багажник
- статистика смены

### 1.3 Владелец

Web dashboard:

- мониторинг активных заказов
- CRM по клиентам
- платежи и рекуррентные списания
- складской учет
- настройки цен
- лог событий и штрафов
- сообщения от клиентов

### 1.4 Backend

Единый backend отвечает за:

- Telegram auth и проверку ролей
- бизнес-логику подписок и заказов
- планировщик задач и уведомлений
- интеграцию с bePaid
- синхронизацию остатков
- историю событий и аудит

## 4. Ключевые сценарии MVP

### 4.1 Подписка клиента

1. Пользователь открывает Web App из Telegram.
2. Принимает оферту и согласие на автосписания.
3. Заполняет телефон и адрес.
4. Настраивает месячный объем и график.
5. Видит расчет.
6. Привязывает карту через bePaid.
7. Получает активную подписку.

### 4.2 Плановая доставка

1. Планировщик создает доставку по графику.
2. За день отправляется напоминание и запрос подтверждения.
3. При подтверждении доставка попадает в очередь.
4. Курьер завершает доставку.
5. Система списывает лимит клиента и складские остатки.

### 4.3 Экспресс-заказ

1. Клиент нажимает дозаказ.
2. Система создает отдельный заказ с полной стоимостью.
3. Курьер принимает заказ.
4. После подтверждения запускается таймер 60 минут.
5. При завершении доставки таймер останавливается.

### 4.4 Изменение графика

1. Клиент запрашивает изменение времени или дней.
2. Система показывает стоимость изменения.
3. После подтверждения выполняется списание.
4. График подписки обновляется и пишется в аудит.

## 5. Технические решения

### 5.1 Frontend

Один `Next.js` проект с тремя route-группами:

- `/app` для клиента
- `/driver` для курьера
- `/owner` для владельца

Так быстрее развивать общий UI-kit, авторизацию и API-клиент.

### 5.2 Backend

Отдельный `Fastify` сервис:

- REST API для web-клиентов
- webhook для Telegram
- webhook для bePaid
- cron-задачи для напоминаний, рекуррентных платежей и таймеров

### 5.3 Хранение

`PostgreSQL` как основная БД.

Критичные журналы:

- оплаты
- штрафы
- изменения графика
- статусы заказов
- складские движения

## 6. Риски и особенности

- интеграция рекуррентных платежей требует отдельного тестового контура bePaid
- Telegram Web App авторизация отличается от обычной web auth
- экспресс-таймер требует точной серверной логики, не только UI
- offline-sync у курьера нужен через локальную очередь событий
- чат владельца с клиентами стоит проектировать как отдельный модуль, чтобы не тормозить MVP
